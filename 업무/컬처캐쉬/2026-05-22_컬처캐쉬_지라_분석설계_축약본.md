# 컬처캐쉬(CULTURE) 결제 연동 — 분석 · 설계

## 현행 진행 상태부터 짚고 가겠습니다

먼저 한 가지 분명히 해두어야 할 부분이 있습니다. 이 문서에서 설명하는 처리 흐름은 두 갈래입니다. 하나는 컬쳐랜드/한국문화진흥과 실제로 통신했을 때의 목표 흐름이고, 다른 하나는 지금 실제로 동작하고 있는 임시 흐름입니다.

현재(2026-05-22) 시점에는 컬쳐랜드 측 가맹점 코드(MemberCode)를 아직 받지 못한 상태입니다. 그래서 컬쳐랜드 로그인 팝업이나 한국문화진흥의 승인·취소 전문(8110/8120/8210/8220) 같은 외부 API를 실제로 호출하지 않습니다. 대신 외부 호출이 일어나야 할 지점들을 핑거페이 내부에서 PASS(우회) 처리하고, 응답값은 고정된 성공값(Mock)으로 합성해서 프로세스가 끊기지 않게 만들어 두었습니다.

그래서 지금 이 모드에서 확인할 수 있는 건 가맹점 진입부터 화면 전이, DB 적재, 결과 통보 인터페이스의 형태까지입니다. 반대로 확인되지 않는 건 실제 인증 정합성, 한국문화진흥의 실 승인/취소 처리, 한도·잔액 차감, 그리고 실거래 결과코드 매핑의 정확성입니다. 가맹점 코드를 받아서 PASS 분기만 제거하면 자동으로 실연동 흐름으로 넘어가도록 구성해 두었습니다.

## 1. 개요

핑거페이에 컬처캐쉬(한국문화진흥, ID/PW 방식)를 신규 결제수단으로 추가하는 프로젝트입니다. 사용자가 컬쳐랜드 로그인 팝업으로 인증하면, 핑거페이 BLD가 한국문화진흥과 승인전문(8110/8120)을 주고받아 결제를 처리하고, 결과를 가맹점에 통보합니다. 이 전 구간이 본 프로젝트의 범위입니다.

| 항목 | 값 |
|---|---|
| 대상 시스템 | FRONT, BLD(v3), DB |
| 결제 종류 | 일반결제, 전체취소|
| 부분취소 | 미지원 |
| 외부 연동 (목표) | 컬쳐랜드(인증), 한국문화진흥(승인/취소) |
| 현재 외부 연동 상태 | 미연동 (PASS 처리, Mock 응답 합성) |
| 개발 전략 | PASS/Mock 우선. 가맹점 MemberCode 수령 전까지 내부 프로세스를 먼저 완성 |
| 비범위 | 부분취소, 분할승인, 컬쳐랜드 부가 API, 별도 백오피스 |

## 2. 영향 범위와 신규 코드값

영향을 받는 시스템과 작업 내용은 다음과 같습니다.

| 시스템 | 영향 |
|---|---|
| FRONT | 컨트롤러 1종 신설, 결제수단 서비스 보강, 매퍼 1종 신설 |
| BLD | 신규 JVM 인스턴스 추가(포트 20501), 처리빈 2종, 매퍼 1종 |
| DB | 신규 테이블 3종, 공통 코드/가맹점 등록 7행 |
| 운영 | 신규 포트(20501) 방화벽 개방, BLD 인스턴스 기동 |

BLD는 결제수단별로 별도 JVM 프로세스로 띄우는 구조라, 컬처캐쉬용 JVM 한 개가 새로 추가됩니다.

신규로 정의한 코드값은 아래와 같습니다.

| 코드 | 값 |
|---|---|
| PM_CD | 32 (값 `CULTURE`) |
| SPM_CD | 07 (값 `CULTURE`) |
| PTN_CD | `CULT` (DB 컬럼 4자 제한으로 인한 예외) |
| 테스트 MID | `100000098m` |
| MOCK MemberCode | `MOCKCULTURE1` (실 코드 수령 시 교체) |
| BLD 포트 | 20501 |

## 3. 데이터 설계

신규로 추가한 테이블은 세 개입니다.

| 테이블 | 위치 | 용도 |
|---|---|---|
| `TBAT_CULTURE` | FRONT | 컬쳐랜드 로그인 인증 결과 |
| `TBTR_CULTURE` | BLD | 거래 원장 (8110/8120 매핑) |
| `TBUS_CULTURE` | BLD | 실패 원장 (8120/8220/8720 실패) |

이 외에 공통 코드 테이블(`TBAD_CODE`)에 PM_CD 32, SPM_CD 07, 결과코드 5종을 등록하고, 가맹점-파트너 매핑(`TBSI_PTN_CPID` / `TBSI_MBS_PTN_LNK` / `TBSI_MBS_SVC`)에 총 7행을 사전 등록합니다. 적용 SQL은 v03 버전으로 TEARDOWN과 CREATE를 모두 포함해 어떤 환경 상태에서도 재실행 가능합니다.

## 4. 처리 흐름

### 4.1 목표 결제 흐름 (실 연동 시, 현재 미가동)

```
가맹점 ─① 결제요청(payMethod=CULTURE)→ FRONT
FRONT ─② TBAT_CULTURE INSERT
FRONT ─③ weblogin.do 팝업→ 컬쳐랜드        ← 실 API
컬쳐랜드 ─④ 인증결과(RetVal/HashNo/잔액/사용금액)→ FRONT
FRONT ─⑤ TBAT_CULTURE UPDATE → ⑥ returnUrl 통보 → 가맹점
가맹점 ─⑦ 승인요청→ FRONT ─⑧ approval(hashString 검증)→ BLD(:20501)
BLD ─⑨ 8110 송신→ 한국문화진흥                ← 실 API
한국문화진흥 ─⑩ 8120 응답→ BLD
BLD ─⑪ TBTR_CULTURE INSERT → ⑫ 승인결과→ FRONT → ⑬ 가맹점
```

### 4.2 목표 취소 흐름 (실 연동 시, 현재 미가동)

전체취소는 가맹점이 취소를 요청하면 FRONT를 거쳐 BLD가 한국문화진흥에 8210 전문을 보내고 8220 응답을 받아 `TBTR_CULTURE`를 갱신하는 흐름입니다. 응답이 실패면 `TBUS_CULTURE`에 적재합니다. 

### 4.3 현재 가동 흐름 (PASS 처리, 임시)

지금 동작하고 있는 모드입니다. 외부 API 호출은 한 건도 발생하지 않습니다. 위 흐름의 외부 호출 지점들을 내부에서 우회하거나 응답을 합성하는 식으로 처리해 두었습니다.

| 단계 | 목표 동작 | 현재 처리 방식 |
|---|---|---|
| ③ 컬쳐랜드 로그인 팝업 | `weblogin.do` 팝업으로 사용자 ID/PW 인증 | 팝업 호출을 건너뛰고, 클라이언트가 곧장 콜백 URL로 form POST |
| ④ 컬쳐랜드 인증 응답 | 컬쳐랜드 서버가 RetVal/HashNo/잔액/사용금액 반환 | 콜백 컨트롤러가 form param과 무관하게 항상 RetVal=0000(성공)으로 합성 |
| ⑨ 한국문화진흥 8110 송신 | BLD → 한국문화진흥 TCP 송신 | 송신하지 않고, BLD 내부에서 8120 성공 응답을 합성 |
| ⑩ 한국문화진흥 8120 응답 | 한국문화진흥 응답 수신 | 고정 성공 응답(Mock) 사용 |
| 8210/8220 (전체취소) | 한국문화진흥 취소 송수신 | 송신하지 않고 Mock 8220 성공 응답 합성 |

이 모드에서 확인되는 것은 가맹점 진입, FRONT 화면 전이, DB 적재(`TBAT_CULTURE`/`TBTR_CULTURE`), 결과 통보 인터페이스의 형태입니다. 

가맹점 코드를 받아 등록한 뒤 `culture/common.html`의 `submitMockReturn()` 분기, `CultureReturnController`의 STEP1 임시 분기, BLD 처리빈의 Mock 합성 호출만 걷어내면 §4.1/4.2 흐름으로 자동 전환됩니다.

## 5. 인증 · 보안

이번 결제수단은 인증 짝이 서로 다른 해시가 두 종류 공존합니다. 한쪽으로 합치는 게 불가능해서 둘 다 살려둬야 합니다.

| 항목 | hashString | LoginHashCode |
|---|---|---|
| 방향 | 가맹점↔핑거페이 | 가맹점↔컬쳐랜드 |
| 알고리즘 | SHA-256 | MD5 |
| 비밀키 | 가맹점 MKEY | 없음 |
| 검증 위치 | 핑거페이 FRONT | 컬쳐랜드 서버 |

현재 상태로 보면, `hashString`은 실제로 생성·검증되고 있습니다. FRONT 자체에서 검증하는 값이라 PASS할 이유가 없습니다. `LoginHashCode`는 코드상 생성되긴 하지만 컬쳐랜드 팝업 자체를 건너뛰고 있어서 외부로 송신되지 않습니다. `TBAT_CULTURE`에 저장만 됩니다.

NONCE 위조방지나 HashNo timeout 강제검증은 아직 구현하지 않았고, 실연동 시점에 보강할 예정입니다.

## 7. 일정 / 인력

마스터 일정은 8주, 2개월 기준입니다.

| 단계 | 기간 |
|---|---|
| 요구사항 확정, DDL 검토 | 2주 |
| FRONT·BLD 개발, DB 적용 | 4주 |
| 통합 테스트(QA) | 2주 |
| 합계 | 8주 (2개월) |

인력은 1인 단독 기준 26.5 MD를 잡았습니다. 분석설계 4.0, FRONT 6.0, BLD 7.0, DB 1.0, 단위테스트 2.0, 통합테스트 2.5, 가맹점 가이드 1.5, 배포 2.5 입니다.

리스크는 한국문화진흥 측 가맹점 계약 일정에 달려 있습니다. MemberCode 발급이 늦어지면 실연동 단계도 그만큼 미뤄집니다.

## 8. 테스트 범위

아래 시나리오는 실연동(§4.1/4.2) 기준으로 작성한 기대 결과입니다. 지금 동작 중인 PASS 모드에서는 일부 항목만 검증이 가능하고, 나머지는 실연동 후에 확인해야 합니다.

| # | 시나리오 | 기대 결과 (실연동 기준) | 현재 PASS 모드 검증 가능 여부 |
|---|---|---|---|
| T1 | 정상 결제 (일반쇼핑) | `TBTR_MSTR.RSLT_CD=0000`, `TBTR_CULTURE` INSERT, 가맹점 통보 | 가능 (Mock 성공 응답) |
| T2 | 정상 결제 (도서공연, BuyBook=Y) | T1 + `TBTR_NTS_CRCT_REQ` INSERT | 가능 |
| T3 | 사용자 인증 취소 (팝업 종료) | `TBAT_CULTURE.AUTH_RSLT_CD=9999`, 가맹점 `resultCd=9999` | 불가 (팝업 SKIP 이라 트리거 불가) |
| T4 | HashNo 만료 후 승인 시도 | `F407`/`F408` 거절 | 불가 (실 HashNo 부재) |
| T5 | 컬쳐랜드 잔액 부족 | `TBUS_CULTURE` INSERT (`CULTURE_RSLT_CD`=잔액부족) | 불가 (실 응답 부재) |
| T6 | 전체취소 정상 | `TBTR_MSTR.TRX_ST_CD=2`, `TBCC_REVOKE_HIST` INSERT | 가능 (Mock 성공 응답) |
